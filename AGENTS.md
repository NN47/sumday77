# AGENTS.md

## Назначение

Этот файл содержит обязательные архитектурные и UX-правила проекта Sumday77.

Перед внесением изменений агент должен:

1. Изучить существующую реализацию затрагиваемого раздела.
2. Не дублировать существующую бизнес-логику.
3. Не создавать локальные исключения, если задачу можно решить универсально.
4. Сохранять единый UX между похожими разделами.
5. После изменений проверить все связанные сценарии.

---

## Общие архитектурные правила

- Использовать стабильные внутренние идентификаторы сущностей.
- Не строить callback data на полном отображаемом русском названии.
- Не дублировать справочники и конфигурации в нескольких обработчиках.
- Общую логику выносить в функции, сервисы или конфигурацию.
- Новые решения должны масштабироваться на будущие типы данных.
- Избегать проверок по конкретным названиям сущностей вида:

```python
if activity.name == "Скакалка":
    ...
```

- Вместо этого использовать свойства, типы и конфигурацию сущности.

## Правила интерфейса Telegram

- Reply-кнопки используются для навигации между разделами и крупными сценариями.
- Inline-кнопки используются для действий над текущим сообщением или объектом.
- Не дублировать одно и то же действие одновременно в reply- и inline-клавиатуре.
- Кнопка «Назад» должна находиться только в одном понятном месте.
- Похожие списки должны использовать одинаковую пагинацию и одинаковые названия кнопок.
- После сохранения, удаления или редактирования обновлять текущий экран и не оставлять устаревшие сообщения, если архитектура позволяет редактировать существующее сообщение.

## Раздел активности

- Шаги являются отдельной сущностью.
- Шаги не должны смешиваться с обычными упражнениями и видами активности.
- Недавние активности формируются из истории пользователя.
- Повторяющиеся активности в списке недавних должны быть уникальными по типу активности.
- Главный экран активности должен показывать конкретные активности за день и их параметры, а не только количество записей.
- Все виды активности должны быть распределены по категориям.
- Категории и поддерживаемые способы ввода должны храниться централизованно.

---

## Универсальные способы ввода активности

Каждая активность должна описывать поддерживаемые способы ввода через конфигурацию или модель.

Примеры:

Бег:
- time
- distance

Скакалка:
- time
- jumps

Отжимания:
- repetitions

Планка:
- time

Правила:

- Если активность поддерживает один способ ввода, сразу открывать соответствующий сценарий.
- Если активность поддерживает несколько способов ввода, сначала показывать выбор способа.
- Интерфейс не должен содержать специальных проверок под конкретное название активности.
- Для нового способа ввода необходимо расширять конфигурацию, а не переписывать общий UI.
- Данные должны сохранять выбранный пользователем способ ввода.
- Отчёты и редактирование должны учитывать фактически использованный способ ввода.

## Форматирование активности

Формат строки должен зависеть от сохранённых параметров:

- Бег — 35 мин (~310 ккал)
- Бег — 5 км (~310 ккал)
- Скакалка — 20 мин (~180 ккал)
- Скакалка — 1 500 прыжков (~180 ккал)
- Отжимания — 40 раз (~18 ккал)
- Жим штанги лёжа — 4 подхода × 10 раз, 60 кг (~45 ккал)

Не выводить параметры, которых у записи нет.

## Перед завершением задачи

Агент должен проверить:

- существующие сценарии не сломаны;
- новая логика не дублирует старую;
- возврат назад работает корректно;
- данные сохраняются и редактируются;
- отчёты отображают правильные единицы;
- старые записи продолжают корректно открываться.
